## Phase 1: 프로젝트 초기 설정 (골격 구축)

**목표**: 개발 환경 완성 및 Notion API 연동 준비

### 📌 할 작업
- [ ] Notion API 키 발급 및 설정
- [ ] `.env.local` 파일 생성 및 환경변수 입력
- [ ] Notion 데이터베이스 생성 (필드: No, Title, Description, Category, CreatedAt, Status)
- [ ] Git 저장소 상태 확인 및 정리

### 🤔 왜 이 순서인가?
모든 개발 작업의 토대가 되므로 가장 먼저 진행합니다. Notion API 연동 없이는 다음 단계로 진행할 수 없습니다.

### ⏱️ 예상 소요 시간
- API 키 발급: 10분
- 데이터베이스 생성: 15분
- 테스트 데이터 입력: 15분
- **소계: 약 40분**

### ✅ 완료 기준
- [x] `.env.local` 파일이 정상 로드됨
- [x] Notion 데이터베이스가 생성됨 (테스트 데이터 5개 이상)
- [x] API 키로 Notion API 호출 성공
- [x] 최신 커밋 푸시 완료

**문제 해결**:
- API 키 401 에러 → Integration 권한 재확인
- 데이터베이스 404 에러 → 데이터베이스 ID 복사본 확인

## Phase 2: 공통 모듈/컴포넌트 개발

**목표**: 재사용 가능한 모듈 및 UI 컴포넌트 준비

### 📌 할 작업

#### 2-1. Notion 클라이언트 유틸리티 (`lib/notion.ts`)
- [ ] `@notionhq/client` 패키지 설치
- [ ] Notion 클라이언트 초기화 함수
- [ ] 데이터베이스 쿼리 함수:
  - `getAllIdeas()` - 모든 아이디어 조회 (No 오름차순)
  - `getIdeaById(id)` - 단일 아이디어 조회
  - `queryIdeasByCategory(category)` - 카테고리별 조회 (향후)
- [ ] TypeScript 인터페이스 정의:
  - `type Idea` - 아이디어 데이터 구조
  - `type IdeaResponse` - API 응답 타입

#### 2-2. UI 컴포넌트
- [ ] `components/IdeaCard.tsx` - 아이디어 카드 컴포넌트
  - Props: 제목, 설명, 카테고리, 상태, 생성일시
  - 스타일: shadcn/ui Card + Badge
- [ ] `components/CategoryBadge.tsx` - 카테고리 배지
  - Props: 카테고리명
  - 색상: 카테고리별 다른 색상
- [ ] `components/StatusBadge.tsx` - 상태 배지
  - Props: 상태명
  - 색상: 상태별 다른 색상

#### 2-3. 유틸리티 및 타입
- [ ] `types/idea.ts` - 전역 타입 정의
- [ ] `lib/date-utils.ts` - 날짜 포매팅 함수
- [ ] `constants/categories.ts` - 카테고리 상수 정의
- [ ] `constants/statuses.ts` - 상태 상수 정의

### 🤔 왜 이 순서인가?
- Notion 클라이언트가 없으면 데이터를 가져올 수 없음
- UI 컴포넌트는 데이터 구조에 의존하므로 타입 정의 후 개발
- 재사용 가능한 모듈을 먼저 만들어 핵심 기능 개발 속도 향상

### ⏱️ 예상 소요 시간
- Notion 클라이언트: 1시간
- UI 컴포넌트: 1시간
- 타입/유틸리티: 30분
- **소계: 약 2.5시간**

### ✅ 완료 기준
- [ ] `lib/notion.ts`의 모든 함수가 TypeScript 타입으로 정의됨
- [ ] `getAllIdeas()` 호출 시 No 순으로 정렬된 배열 반환 확인
- [ ] `IdeaCard` 컴포넌트가 Storybook 또는 테스트 페이지에서 렌더링 확인
- [ ] 모든 컴포넌트에 한국어 주석 추가됨
- [ ] `npm run lint` 통과 (타입 에러 없음)

**테스트 방법**:
```bash
# 콘솔에서 Notion API 호출 테스트
node -e "require('./lib/notion.ts').getAllIdeas().then(console.log)"
```

## Phase 4: 추가 기능 & 최적화 (MVP 이후)

**목표**: 사용자 경험 향상 및 성능 최적화

### 📌 할 작업 (우선순위 순)

#### 4-1. 성능 최적화 (높음)
- [ ] ISR (Incremental Static Regeneration) 설정
  - 목록 페이지: 매 10분마다 재생성
  - 상세 페이지: 동적 콘텐츠는 on-demand ISR
- [ ] 이미지 최적화 (있을 경우)
- [ ] 번들 크기 최적화
  - `npm run analyze` 로 번들 분석
- [ ] Lighthouse 성능 점수 목표: 90 이상

#### 4-2. 필터링 기능 (중간)
- [ ] 카테고리별 필터 UI
- [ ] 상태별 필터 UI
- [ ] 쿼리 파라미터 기반 필터링 (`?category=frontend&status=진행중`)

#### 4-3. 검색 기능 (중간)
- [ ] 제목/내용 검색 인풋
- [ ] 클라이언트 사이드 검색 (Notion 데이터 로드 후)
- [ ] 또는 Notion Query Database 필터 활용

#### 4-4. 접근성 개선 (중간)
- [ ] ARIA 라벨 추가
- [ ] 키보드 네비게이션 지원
- [ ] 스크린 리더 테스트

#### 4-5. 분석 & 모니터링 (낮음)
- [ ] Google Analytics 또는 Vercel Analytics 설정
- [ ] 에러 추적 (Sentry 등)

#### 4-6. 배포 및 CI/CD (낮음)
- [ ] GitHub Actions 워크플로우 설정
  - 푸시 시 자동 테스트 & 린트
  - PR 생성 시 자동 빌드 체크
- [ ] Vercel 배포 설정
- [ ] 커스텀 도메인 설정 (선택)

### 🤔 왜 이 순서인가?
- 성능 최적화는 사용자 경험에 직접 영향
- 필터링, 검색은 데이터가 많아질 때 필요
- 접근성은 모든 사용자를 배려하는 기본
- 배포는 프로덕션 준비 단계

### ⏱️ 예상 소요 시간
- 성능 최적화: 2시간
- 필터링 & 검색: 2시간
- 접근성 & 모니터링: 1.5시간
- CI/CD & 배포: 1시간
- **소계: 약 6.5시간** (병렬 진행 가능)

### ✅ 완료 기준
- [ ] Lighthouse 성능 점수 90 이상
- [ ] 필터링 & 검색이 정상 동작
- [ ] 키보드만으로 전체 사이트 네비게이션 가능
- [ ] Vercel에 배포되고 공개 URL 접근 가능
- [ ] GitHub Actions 워크플로우 통과
